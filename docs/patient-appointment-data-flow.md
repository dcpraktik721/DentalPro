# Формирование данных записи пациента

Дата фиксации: `2026-02-23`.

Источник фактов:
`artifacts/appointment/appointment_flow_scan.json`

## 1. Основные модули потока
- Расписание: `/visits/schedule/index` (фактически открывает `/visits/schedule/week`)
- Отложенная запись: `/holdrecord/ajax.json?action=add`
- База пациентов: `/cbase/index.html`
- Записи страховых пациентов: `/insurance/schedule`
- Служебные формы расписания:
  - создание интервала: `/stompro/scheduler/formAddInterval?...`
  - редактирование интервала/записи: `/stompro/scheduler/formEditInterval?recordID=...`

## 2. Что формирует саму запись

### 2.1. Форма отложенной записи (подтверждено)
Форма отправляется на:
- `POST /holdrecord/ajax.json`

Ключевые поля:
- `action=add_save`
- `client` (идентификатор пациента)
- `date_visit` (желаемая дата визита)
- `date` (дата обзвона)
- `department_id` (направление/отделение)
- `type` (тип визита)
- `doctor` (врач)
- `comment` (комментарий)

Наблюдаемые справочники в форме:
- Отделения: Хирурги, Терапевты, Ортодонты, Ассистенты, Ортопеды, Пародонтологи, Анестезиологи, Детская терапия, Гигиенисты, Рентген лаборант.
- Типы записи: консультация, лечение, проф. осмотр, проф. гигиена и др. (список задается справочником).

### 2.2. Поиск/подтяжка пациента в момент записи
Обнаруженные endpoint hints:
- `/cbase/search.json?showall=1`
- `/cbase/search.html?display_name=`

Это указывает, что при создании записи пациент подбирается через модуль базы пациентов (`cbase`) и затем подставляется в поле `client`.

## 3. Где хранятся и как фильтруются данные записи

### 3.1. Фильтры в базе пациентов
Форма фильтрации в `/cbase/index.html` содержит поля:
- `record_date[start]`, `record_date[end]`
- `doctor_id`, `manager`, `service_id`, `service_directory_id[]`
- `phone`, `birthday`, `sex`
- `client_group_ids[]`, `company_id`
- `columns[]` (набор выводимых столбцов)

Вывод: данные записи пациента используются как часть среза базы пациентов и доступны через фильтрацию по дате, врачу, услуге, координатору.

### 3.2. Страховые записи
`/insurance/schedule` использует параметр `date` и формирует отдельный срез расписания для ДМС.

## 4. Жизненный цикл записи (операционная модель)

1. Создание/подбор пациента
- через `cbase` (поиск/карточка пациента), либо через сценарий «новый пациент в расписании».

2. Создание записи
- через форму отложенной записи (`holdrecord`) с выбором даты, отделения, типа, врача.

3. Привязка к расписанию врача
- через `stompro/scheduler/formAddInterval` (слот/интервал) и последующее редактирование (`formEditInterval`).

4. Контроль исполнения
- отображение в расписании, списках пациентов/VIP, страховых списках и отчетах по приемам.

## 6. Практический вывод для интеграции
Для устойчивой интеграции записи пациента минимальный payload должен включать:
- идентификатор пациента (`client`)
- дата/время визита (`date_visit` + слот/интервал)
- врач (`doctor`)
- отделение (`department_id`)
- тип визита (`type`)
- служебный комментарий (`comment`)

И отдельно нужен контур справочников (врачи, отделения, типы записи), иначе запись нельзя валидно построить.
